# آموزش ساده Big O Notation

اگر برنامه‌نویسی، الگوریتم، ساختمان داده یا مصاحبه فنی رو شروع کرده باشی، احتمالاً اسم Big O رو زیاد شنیدی.

اولش ممکنه ترسناک به نظر بیاد، ولی اصل ماجرا خیلی ساده‌تر از چیزیه که فکر می‌کنی.

مفهوم Big O فقط کمک می‌کنه بفهمیم وقتی داده‌ها زیاد می‌شن، کدمون چطور رفتار می‌کنه.

یعنی آیا هنوز خوب و سریع کار می‌کنه؟
یا با زیاد شدن داده‌ها، کم‌کم کند و سنگین می‌شه؟

---

## مفهوم Big O یعنی چی؟

مفهوم Big O روشی برای توضیح دادن رشد زمان اجرا یا مصرف حافظه یک الگوریتمه.

به زبان ساده:

مفهوم Big O بهمون می‌گه وقتی اندازه ورودی بزرگ‌تر می‌شه، مقدار کاری که کد باید انجام بده چطور رشد می‌کنه.

نکته مهم:

مفهوم Big O معمولاً نمی‌گه کد دقیقاً چند ثانیه طول می‌کشه.

مثلاً نمی‌گه این تابع دقیقاً ۲ میلی‌ثانیه یا ۵ ثانیه طول می‌کشه.

این مفهوم بیشتر می‌گه:

اگر تعداد داده‌ها زیاد شد، زمان اجرا چه شکلی رشد می‌کنه؟

مثلاً اگر تعداد داده‌ها دو برابر بشه:

* زمان اجرا تقریباً همون می‌مونه؟
* دو برابر می‌شه؟
* چهار برابر می‌شه؟
* یا خیلی سریع‌تر از چیزی که انتظار داریم زیاد می‌شه؟

مفهوم Big O دقیقاً برای همین سؤال‌هاست.

---

## چرا مفهوم Big O مهمه؟

وقتی برنامه‌ات فقط با ۱۰ تا داده کار می‌کنه، تقریباً هر راه‌حلی ممکنه خوب به نظر برسه.

ولی وقتی داده‌ها بشن هزار تا، یک میلیون تا، یا وقتی اپلیکیشن به تعداد زیادی کاربر برسه، دیگه فرق الگوریتم‌ها خیلی مهم می‌شه.

ممکنه یک کدی برای ۱۰۰ تا آیتم عالی کار کنه، ولی برای یک میلیون آیتم سرور رو تحت فشار بذاره.

پس مفهوم Big O کمک می‌کنه قبل از اینکه سیستم در مقیاس بزرگ به مشکل بخوره، بفهمیم راه‌حلی که نوشتیم چقدر قابل اتکاست.

---

# مثال ساده: پیدا کردن دوستت در استادیوم

فرض کن وارد یک استادیوم خیلی بزرگ شدی و دنبال دوستت می‌گردی.

---

## حالت اول: هیچ اطلاعاتی نداری

اگر ندونی دوستت کجا نشسته، باید یکی‌یکی صندلی‌ها رو چک کنی.

اگر استادیوم `n` تا صندلی داشته باشه، در بدترین حالت ممکنه مجبور بشی همه اون `n` تا صندلی رو بررسی کنی.

اینجا پیچیدگی زمانی می‌شه `O(n)`.

به این می‌گیم زمان خطی یا Linear Time.

چرا؟

چون هرچقدر تعداد صندلی‌ها بیشتر بشه، زمان جست‌وجو هم تقریباً به همون نسبت بیشتر می‌شه.

مثلاً:

* اگر ۱۰۰ صندلی باشه، شاید مجبور شی تا ۱۰۰ صندلی رو چک کنی
* اگر ۱۰۰۰ صندلی باشه، شاید مجبور شی تا ۱۰۰۰ صندلی رو چک کنی
* اگر یک میلیون صندلی باشه، شاید مجبور شی تا یک میلیون صندلی رو چک کنی

پس مقدار کار با تعداد صندلی‌ها رشد می‌کنه.

---

## حالت دوم: شماره دقیق صندلی رو داری

حالا فرض کن دوستت بهت گفته:

من ردیف H، صندلی D110 نشستم.

اینجا دیگه لازم نیست کل استادیوم رو بگردی.

مستقیم می‌ری سراغ همون ردیف و همون صندلی.

اینجا فرقی نمی‌کنه استادیوم ۱۰۰ صندلی داشته باشه یا یک میلیون صندلی.

چون تو از قبل آدرس دقیق رو داری.

پس تعداد قدم‌های منطقی برای پیدا کردنش تقریباً ثابت می‌مونه.

اینجا پیچیدگی زمانی می‌شه `O(1)`.

به این می‌گیم زمان ثابت یا Constant Time.

---

## معنی `n` چیه؟

در Big O، معمولاً `n` یعنی اندازه ورودی.

مثلاً:

* در یک لیست، `n` یعنی تعداد آیتم‌های لیست
* در یک آرایه، `n` یعنی تعداد عناصر آرایه
* در یک جدول دیتابیس، `n` می‌تونه تعداد رکوردها باشه
* در مثال استادیوم، `n` یعنی تعداد صندلی‌ها

پس وقتی می‌گیم یک الگوریتم `O(n)` است، یعنی مقدار کارش با تعداد ورودی‌ها رشد می‌کنه.

---

# پیچیدگی‌های رایج در Big O

حالا بریم سراغ حالت‌های معروفی که زیاد می‌بینی.

---

## پیچیدگی `O(1)` — زمان ثابت

در پیچیدگی `O(1)`، زمان اجرا با زیاد شدن ورودی تغییر خاصی نمی‌کنه.

یعنی الگوریتم تقریباً یک مقدار ثابت کار انجام می‌ده.

مثلاً گرفتن اولین آیتم یک لیست:

</div>

```python
def get_first_item(items):
    return items[0]
```

<div dir="rtl" align="right">

اینجا مهم نیست لیست ۱۰ آیتم داره یا ۱۰ میلیون آیتم.

ما فقط می‌ریم سراغ اولین آیتم.

پس پیچیدگی زمانی این کد `O(1)` است.

نکته مهم:

پیچیدگی `O(1)` یعنی نسبت به اندازه ورودی ثابت می‌مونه.
نه اینکه همیشه در دنیای واقعی instant یا جادویی سریع باشه.

ممکنه یک عملیات ثابت خودش سنگین باشه، ولی با زیاد شدن تعداد داده‌ها بیشتر نمی‌شه.

---

## پیچیدگی `O(log n)` — زمان لگاریتمی

در پیچیدگی `O(log n)`، معمولاً در هر مرحله بخش بزرگی از داده‌ها حذف می‌شه.

معروف‌ترین مثالش الگوریتم Binary Search یا جست‌وجوی دودوییه.

فرض کن یک لیست مرتب‌شده داری.

به جای اینکه از اول تا آخر لیست رو بگردی، از وسط شروع می‌کنی.

اگر عدد مورد نظر کوچک‌تر از مقدار وسط باشه، نصف سمت راست رو کنار می‌ذاری.

اگر بزرگ‌تر باشه، نصف سمت چپ رو کنار می‌ذاری.

یعنی هر بار نصف داده‌ها حذف می‌شن.

</div>

```python
def binary_search(items, target):
    left = 0
    right = len(items) - 1

    while left <= right:
        middle = (left + right) // 2

        if items[middle] == target:
            return middle

        if items[middle] < target:
            left = middle + 1
        else:
            right = middle - 1

    return -1
```

<div dir="rtl" align="right">

پیچیدگی زمانی این الگوریتم `O(log n)` است.

نکته مهم:

الگوریتم Binary Search فقط وقتی درست کار می‌کنه که داده‌ها مرتب‌شده باشن.

---

## پیچیدگی `O(n)` — زمان خطی

در پیچیدگی `O(n)`، مقدار کار الگوریتم همراه با تعداد ورودی‌ها رشد می‌کنه.

مثلاً جست‌وجوی یک آیتم در لیست:

</div>

```python
def contains(items, target):
    for item in items:
        if item == target:
            return True

    return False
```

<div dir="rtl" align="right">

در بدترین حالت، ممکنه آیتم مورد نظر آخر لیست باشه یا اصلاً داخل لیست نباشه.

پس مجبور می‌شیم کل لیست رو بررسی کنیم.

اگر لیست `n` آیتم داشته باشه، در بدترین حالت تا `n` بررسی انجام می‌دیم.

پس پیچیدگی زمانی این کد `O(n)` است.

مثال‌های رایج:

* پیمایش کامل یک لیست
* پیدا کردن بزرگ‌ترین عدد در یک لیست
* محاسبه مجموع آیتم‌ها
* فیلتر کردن آیتم‌ها بر اساس یک شرط

---

## پیچیدگی `O(n log n)`

این پیچیدگی رو زیاد در الگوریتم‌های مرتب‌سازی خوب می‌بینی.

مثلاً الگوریتم Merge Sort معمولاً پیچیدگی زمانی `O(n log n)` دارد.

ایده کلی اینه که داده‌ها چند بار تقسیم می‌شن و بعد دوباره ترکیب می‌شن.

یک نمونه مفهومی:

</div>

```python
def merge_sort(items):
    if len(items) <= 1:
        return items

    middle = len(items) // 2

    left = merge_sort(items[:middle])
    right = merge_sort(items[middle:])

    return merge(left, right)
```

<div dir="rtl" align="right">

نکته مهم:

خیلی وقت‌ها می‌شنوی الگوریتم Quick Sort هم `O(n log n)` است.

این حرف برای حالت میانگین درسته، ولی Quick Sort در بدترین حالت می‌تونه `O(n²)` بشه.

پس بهتره دقیق‌تر بگیم:

* الگوریتم Merge Sort در حالت میانگین و بدترین حالت معمولاً `O(n log n)` است
* الگوریتم Quick Sort در حالت میانگین معمولاً `O(n log n)` است، ولی بدترین حالتش می‌تونه `O(n²)` باشه

---

## پیچیدگی `O(n²)` — زمان درجه دو

در پیچیدگی `O(n²)` معمولاً با دو حلقه تو در تو روبه‌رو هستیم.

یعنی برای هر آیتم، دوباره چند آیتم دیگر رو هم بررسی می‌کنیم.

مثلاً بررسی آیتم تکراری با روش ساده:

</div>

```python
def has_duplicate(items):
    for i in range(len(items)):
        for j in range(i + 1, len(items)):
            if items[i] == items[j]:
                return True

    return False
```

<div dir="rtl" align="right">

اینجا برای هر آیتم، چند آیتم دیگر هم بررسی می‌شه.

اگر تعداد آیتم‌ها زیاد بشه، تعداد مقایسه‌ها خیلی سریع بالا می‌ره.

مثلاً:

* اگر `n = 10` باشه، حدوداً ۱۰۰ مقایسه ممکنه داشته باشیم
* اگر `n = 100` باشه، حدوداً ۱۰ هزار مقایسه ممکنه داشته باشیم
* اگر `n = 1000` باشه، حدوداً یک میلیون مقایسه ممکنه داشته باشیم

پس پیچیدگی `O(n²)` برای داده‌های بزرگ می‌تونه خیلی سنگین بشه.

---

## پیچیدگی `O(2^n)` — زمان نمایی

در پیچیدگی `O(2^n)`، با اضافه شدن هر ورودی جدید، تعداد حالت‌ها خیلی سریع رشد می‌کنه.

این حالت معمولاً وقتی دیده می‌شه که الگوریتم تعداد زیادی حالت ممکن رو بررسی می‌کنه.

مثال معروفش نسخه ساده و خام Fibonacci با recursion است:

</div>

```python
def fibonacci(n):
    if n <= 1:
        return n

    return fibonacci(n - 1) + fibonacci(n - 2)
```

<div dir="rtl" align="right">

در این روش، خیلی از محاسبات چندین بار تکرار می‌شن.

برای همین با بزرگ شدن `n`، زمان اجرا خیلی سریع زیاد می‌شه.

نکته:

برای آموزش، معمولاً این مثال رو به عنوان نمونه‌ای از رشد نمایی توضیح می‌دن.
در عمل می‌شه با memoization یا dynamic programming همین مسئله رو خیلی بهتر حل کرد.

---

## پیچیدگی `O(n!)` — زمان فاکتوریل

پیچیدگی `O(n!)` یکی از سنگین‌ترین حالت‌های رایجه.

این حالت معمولاً وقتی پیش میاد که بخوای همه ترتیب‌های ممکن رو بررسی کنی.

مثلاً فرض کن ۵ نفر داریم و می‌خوایم همه حالت‌های ممکن نشستن اون‌ها رو بررسی کنیم.

تعداد حالت‌ها می‌شه `5! = 120`.

ولی اگر تعداد افراد بشه ۱۰، تعداد حالت‌ها می‌شه `10! = 3,628,800`.

و اگر تعداد افراد بیشتر بشه، تعداد حالت‌ها خیلی سریع غیرقابل کنترل می‌شه.

مثال‌های رایج:

* تولید همه permutationها
* بررسی همه ترتیب‌های ممکن
* بعضی روش‌های brute force برای مسئله‌هایی مثل Traveling Salesman

---

# ترتیب کلی از بهتر به بدتر

به شکل ساده، معمولاً این ترتیب رو در ذهن داشته باش:

</div>

```text
O(1)
O(log n)
O(n)
O(n log n)
O(n²)
O(2^n)
O(n!)
```

<div dir="rtl" align="right">

یعنی معمولاً `O(1)` بهتر از `O(n)` است، و `O(n)` معمولاً خیلی بهتر از `O(n²)` است.

البته در دنیای واقعی فقط Big O مهم نیست.

گاهی اندازه داده‌ها کوچک است، یا کد ساده‌تر ارزش بیشتری دارد، یا هزینه حافظه مهم‌تر است.

ولی برای فهم عملکرد در مقیاس بزرگ، Big O ابزار خیلی مهمیه.

---

# چند نکته رایج درباره ساختمان داده‌ها

این‌ها چیزهایی هستن که در مصاحبه‌ها و پروژه‌ها زیاد به کارت میان.

---

## آرایه یا لیست

اگر index رو داشته باشی، دسترسی به یک آیتم معمولاً `O(1)` است.

مثلاً:

</div>

```python
item = items[5]
```

<div dir="rtl" align="right">

ولی اگر بخوای دنبال یک مقدار خاص بگردی، معمولاً باید لیست رو پیمایش کنی.

پس جست‌وجوی ساده در لیست معمولاً `O(n)` است.

همچنین اضافه یا حذف کردن از وسط لیست می‌تونه `O(n)` باشه، چون ممکنه آیتم‌های بعدی جابه‌جا بشن.

---

## دیکشنری یا Hash Map

در Python، دیکشنری برای پیدا کردن مقدار با key معمولاً در حالت میانگین `O(1)` است.

مثلاً:

</div>

```python
user = users_by_id["123"]
```

<div dir="rtl" align="right">

ولی نکته دقیق‌تر اینه که این حالت میانگینه.

در شرایط خاص و بدترین حالت، عملیات روی hash table می‌تونه سنگین‌تر بشه.

برای سطح آموزشی و مصاحبه‌های معمولی، اینکه dictionary lookup رو معمولاً `O(1)` بدونی کافیه، ولی بهتره بدونی این حرف مربوط به average case است.

---

## لیست پیوندی یا Linked List

در ساختار Linked List، اگر به خود node دسترسی داشته باشی، حذف یا اضافه کردن می‌تونه `O(1)` باشه.

ولی اگر بخوای دنبال یک آیتم خاص بگردی، باید از اول لیست حرکت کنی.

پس جست‌وجو یا دسترسی به آیتم خاص معمولاً `O(n)` است.

این نکته مهمه، چون خیلی‌ها اشتباه فکر می‌کنن Linked List همیشه برای insert و delete بهتره.

واقعیت اینه:

اگر node رو از قبل داشته باشی، خوبه.
ولی اگر اول باید بگردی و پیداش کنی، اون جست‌وجو خودش هزینه داره.

---

## درخت‌های متعادل

درخت‌های متعادل مثل AVL Tree، Red-Black Tree یا B-Tree معمولاً برای عملیات‌هایی مثل search، insert و delete پیچیدگی `O(log n)` دارن.

دلیلش اینه که ساختار درخت اجازه می‌ده هر بار بخش بزرگی از داده‌ها کنار گذاشته بشه.

در دیتابیس‌ها هم indexها معمولاً از ساختارهایی مثل B-Tree یا ساختارهای مشابه استفاده می‌کنن تا جست‌وجو سریع‌تر بشه.

البته در دیتابیس واقعی، فقط Big O مهم نیست.
دیسک، کش، query planner، اندازه داده و نوع query هم تأثیر دارن.

---

# چطور Big O رو تشخیص بدیم؟

چند قانون ساده وجود داره که برای شروع خیلی کمک می‌کنه.

---

## قانون اول: عملیات مستقیم معمولاً `O(1)` است

مثلاً:

</div>

```python
x = items[0]
```

<div dir="rtl" align="right">

یا:

</div>

```python
user = users_by_id["123"]
```

<div dir="rtl" align="right">

اگر مستقیم به یک مقدار دسترسی داشته باشی، معمولاً با `O(1)` طرفی.

---

## قانون دوم: یک حلقه روی ورودی معمولاً `O(n)` است

مثلاً:

</div>

```python
for item in items:
    print(item)
```

<div dir="rtl" align="right">

اگر حلقه به اندازه تعداد آیتم‌ها اجرا بشه، معمولاً پیچیدگی زمانی `O(n)` است.

---

## قانون سوم: دو حلقه تو در تو معمولاً `O(n²)` هستند

مثلاً:

</div>

```python
for item1 in items:
    for item2 in items:
        print(item1, item2)
```

<div dir="rtl" align="right">

اینجا برای هر آیتم، دوباره همه آیتم‌ها بررسی می‌شن.

پس اگر `n` تا آیتم داشته باشیم، تعداد عملیات تقریباً می‌شه `n × n`.

یعنی پیچیدگی زمانی می‌شه `O(n²)`.

---

## قانون چهارم: نصف کردن داده‌ها معمولاً نشانه `O(log n)` است

اگر الگوریتم در هر مرحله نصف داده‌ها رو حذف کنه، احتمالاً پیچیدگی زمانی `O(log n)` داره.

مثل Binary Search.

---

## قانون پنجم: ضرایب ثابت رو حذف می‌کنیم

فرض کن این کد رو داریم:

</div>

```python
def print_items_twice(items):
    for item in items:
        print(item)

    for item in items:
        print(item)
```

<div dir="rtl" align="right">

این کد دو بار روی لیست می‌چرخه.

از نظر دقیق، حدوداً `2n` عملیات داریم.

ولی در Big O ضریب ثابت رو حذف می‌کنیم.

پس می‌گیم `O(n)`، نه `O(2n)`.

چون Big O بیشتر به شکل رشد اهمیت می‌ده، نه تعداد دقیق عملیات.

---

## قانون ششم: بخش کندتر مهم‌تره

فرض کن این کد رو داریم:

</div>

```python
def example(items):
    print(items[0])

    for item in items:
        print(item)

    for item1 in items:
        for item2 in items:
            print(item1, item2)
```

<div dir="rtl" align="right">

اینجا سه بخش داریم:

* بخش ثابت با پیچیدگی `O(1)`
* یک حلقه ساده با پیچیدگی `O(n)`
* دو حلقه تو در تو با پیچیدگی `O(n²)`

وقتی ورودی بزرگ بشه، بخش `O(n²)` از همه بیشتر اثر می‌ذاره.

پس پیچیدگی کلی این کد رو معمولاً می‌گیم `O(n²)`.

---

# چند مثال کاربردی با Python

## مثال زمان ثابت `O(1)`

</div>

```python
def get_user_by_id(users_by_id, user_id):
    return users_by_id[user_id]
```

<div dir="rtl" align="right">

اگر `users_by_id` یک dictionary باشه، گرفتن user با key معمولاً در حالت میانگین `O(1)` است.

---

## مثال زمان خطی `O(n)`

</div>

```python
def find_user(users, target_email):
    for user in users:
        if user["email"] == target_email:
            return user

    return None
```

<div dir="rtl" align="right">

اینجا ممکنه مجبور شیم همه کاربران رو یکی‌یکی بررسی کنیم.

پس پیچیدگی زمانی این کد `O(n)` است.

---

## بهتر کردن مثال قبلی

اگر به جای لیست، از dictionary استفاده کنیم:

</div>

```python
users_by_email = {
    "ali@example.com": {"name": "Ali"},
    "sara@example.com": {"name": "Sara"},
}

def find_user(users_by_email, target_email):
    return users_by_email.get(target_email)
```

<div dir="rtl" align="right">

در این حالت پیدا کردن کاربر با email معمولاً در حالت میانگین `O(1)` است.

این دقیقاً همون چیزیه که در پروژه‌های واقعی اهمیت پیدا می‌کنه.

گاهی با انتخاب درست ساختمان داده، می‌تونی یک عملیات کند رو خیلی سریع‌تر کنی.

---

## مثال زمان درجه دو `O(n²)`

</div>

```python
def find_pairs(numbers):
    pairs = []

    for a in numbers:
        for b in numbers:
            pairs.append((a, b))

    return pairs
```

<div dir="rtl" align="right">

اینجا برای هر عدد، دوباره همه عددها بررسی می‌شن.

اگر ۱۰۰ عدد داشته باشیم، حدوداً ۱۰ هزار جفت ساخته می‌شه.

پس پیچیدگی زمانی این کد `O(n²)` است.

---

# پیچیدگی زمانی و پیچیدگی فضایی

تا اینجا بیشتر درباره زمان اجرا حرف زدیم.

ولی Big O فقط برای زمان نیست.

گاهی درباره حافظه مصرفی هم حرف می‌زنیم.

---

## پیچیدگی زمانی

پیچیدگی زمانی یعنی:

وقتی اندازه ورودی بزرگ‌تر می‌شه، زمان اجرای الگوریتم چطور رشد می‌کنه؟

مثلاً `O(n)` یعنی زمان اجرا با تعداد ورودی‌ها رشد می‌کنه.

---

## پیچیدگی فضایی

پیچیدگی فضایی یعنی:

وقتی اندازه ورودی بزرگ‌تر می‌شه، مقدار حافظه اضافه‌ای که الگوریتم نیاز داره چطور رشد می‌کنه؟

مثلاً این کد یک لیست جدید می‌سازه:

</div>

```python
def double_items(items):
    result = []

    for item in items:
        result.append(item * 2)

    return result
```

<div dir="rtl" align="right">

چون به اندازه تعداد آیتم‌ها یک لیست جدید می‌سازیم، پیچیدگی فضایی این کد `O(n)` است.

اما این کد فقط یک متغیر کمکی داره:

</div>

```python
def sum_items(items):
    total = 0

    for item in items:
        total += item

    return total
```

<div dir="rtl" align="right">

زمان اجرای این کد `O(n)` است، چون باید همه آیتم‌ها رو ببینه.

ولی حافظه اضافه‌ای که مصرف می‌کنه `O(1)` است، چون فقط یک متغیر `total` داریم.

---

# اشتباهات رایج درباره Big O

## اشتباه اول: Big O یعنی زمان دقیق اجرا

نه.

مفهوم Big O زمان دقیق اجرا رو نمی‌گه.

ممکنه یک کد `O(n)` برای ورودی کوچک از یک کد `O(1)` در عمل سریع‌تر باشه.

مفهوم Big O بیشتر برای اینه که بفهمیم وقتی ورودی بزرگ می‌شه، رشد زمان اجرا چه شکلیه.

---

## اشتباه دوم: هر چیزی که `O(1)` است همیشه سریع است

نه لزوماً.

پیچیدگی `O(1)` یعنی با زیاد شدن اندازه ورودی، مقدار کار رشد نمی‌کنه.

ولی ممکنه همون کار ثابت، خودش سنگین باشه.

پس `O(1)` یعنی constant نسبت به ورودی، نه همیشه فوق‌سریع.

---

## اشتباه سوم: همیشه باید بهترین Big O رو انتخاب کنیم

در پروژه واقعی، فقط Big O مهم نیست.

گاهی خوانایی کد، سادگی، زمان توسعه، مصرف حافظه و نیاز واقعی پروژه هم مهمه.

اگر لیست تو فقط ۲۰ آیتم داره، شاید یک راه‌حل ساده `O(n)` کاملاً کافی باشه.

ولی اگر قراره با میلیون‌ها رکورد کار کنی، باید خیلی جدی‌تر به پیچیدگی فکر کنی.

---

## اشتباه چهارم: Big O همیشه درباره بدترین حالت است

در خیلی از مصاحبه‌ها، وقتی فقط می‌گن complexity، معمولاً منظورشون worst case یا بدترین حالته.

ولی همیشه باید حواست باشه که گاهی داریم درباره حالت‌های مختلف حرف می‌زنیم:

* بهترین حالت یا Best Case
* حالت میانگین یا Average Case
* بدترین حالت یا Worst Case

مثلاً درباره Quick Sort، حالت میانگین و بدترین حالت می‌تونن متفاوت باشن.

پس اگر لازم شد، دقیق بگو داری درباره کدوم حالت صحبت می‌کنی.

---

# یک روش سریع برای فکر کردن به Big O

از خودت بپرس:

اگر تعداد ورودی‌ها دو برابر بشه، مقدار کار چی می‌شه؟

* اگر تقریباً همون بمونه، احتمالاً `O(1)` است
* اگر خیلی کم بیشتر بشه، احتمالاً `O(log n)` است
* اگر تقریباً دو برابر بشه، احتمالاً `O(n)` است
* اگر بیشتر از خطی رشد کنه، ممکنه `O(n log n)` باشد
* اگر حدوداً چهار برابر بشه، احتمالاً `O(n²)` است
* اگر خیلی سریع منفجر بشه، ممکنه `O(2^n)` یا `O(n!)` باشه

---

# چطوری توی مصاحبه توضیحش بدیم؟

اگر در مصاحبه ازت پرسیدن Big O چیه، می‌تونی ساده بگی:

مفهوم Big O روشی برای توضیح دادن اینه که وقتی اندازه ورودی زیاد می‌شه، زمان اجرا یا حافظه مصرفی الگوریتم چطور رشد می‌کنه.

بعد یک مثال بزن:

مثلاً اگر دنبال یک نفر توی استادیوم بگردم و هیچ اطلاعاتی نداشته باشم، شاید مجبور بشم همه صندلی‌ها رو چک کنم. این می‌شه `O(n)`.

ولی اگر شماره دقیق ردیف و صندلی رو داشته باشم، مستقیم می‌رم همون‌جا. این می‌شه `O(1)`.

بعد هم اضافه کن:

مفهوم Big O زمان دقیق رو نمی‌گه؛ فقط شکل رشد رو نشون می‌ده.

همین جواب برای شروع خیلی خوبه.

---

# خلاصه خیلی کوتاه

مفهوم Big O کمک می‌کنه بفهمیم کدمون وقتی داده‌ها زیاد می‌شن، چقدر خوب دوام میاره.

به زبان ساده:

* پیچیدگی `O(1)` یعنی تقریباً ثابت
* پیچیدگی `O(log n)` یعنی خیلی آروم رشد می‌کنه
* پیچیدگی `O(n)` یعنی همراه با ورودی رشد می‌کنه
* پیچیدگی `O(n log n)` معمولاً برای مرتب‌سازی‌های خوب دیده می‌شه
* پیچیدگی `O(n²)` یعنی برای داده‌های بزرگ سریع سنگین می‌شه
* پیچیدگی `O(2^n)` و `O(n!)` یعنی برای ورودی‌های بزرگ معمولاً خطرناک و غیرعملی می‌شن

اگر فقط یک جمله بخوای به خاطر بسپاری:

مفهوم Big O یعنی بفهمیم وقتی مقیاس بزرگ می‌شه، کدمون چقدر خوب یا بد عمل می‌کنه.
